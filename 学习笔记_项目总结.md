# 项目学习笔记

## 一、选择原因

选择这个项目是因为想更多地尝试感兴趣且与专业相关的内容。我学的是集成电路设计与集成系统，平时接触硬件比较多，而 CPU 微架构性能分析正好是硬件与软件的交汇点——从芯片的 Cache 结构、分支预测器到操作系统调度，再到上层 profiling 工具的使用，整个链路让我能从更底层理解计算机系统。

## 二、项目难点

1. **环境配置**：VMware 虚拟机下 perf 的 PMU 硬件计数器默认不可用，需要在宿主机修改 .vmx 文件开启透传，且 AMD CPU 在 VMware 中透传效果有限，最终 instruction、cache-misses 等硬件事件仍无法正常采集。

2. **数据分析**：perf stat 输出的指标（IPC、Cache Miss Rate、TLB Miss Rate 等）需要结合 CPU 微架构原理解释，哪些指标能反应什么问题，5 种负载之间如何横向对比，这些都需要对处理器流水线有一定理解。

3. **Docker + perf 集成**：容器内 perf 版本需与宿主机内核匹配，否则 perf record 无法正常工作，同时缺少的共享库也需要手动安装。

## 三、负责的事情

主要从硬件角度出发做性能分析。包括：
- 使用 perf stat 采集五类负载的 CPU 微架构指标
- 生成火焰图并分析热点函数（实际函数名显示需从源码编译 stress-ng）
- 编写 C 语言程序验证 Cache Line 大小对内存访问性能的影响
- 搭建 Docker 容器化的持续 CPU Profiling 工具

由于时间有限（共约 5 天），没有按部就班地系统学习，而是以任务驱动的方式边做边学：

## 六、学习心得

在做项目的过程中遇到了不少问题，每一次排查和解决的过程都让我对 CPU 性能分析有了更深的理解。

### 问题 1：perf stat 硬件事件全部为 0

- **事实（空）**：在 Ubuntu 虚拟机中运行 `perf stat`，instructions、cache-misses 等指标全部显示为 0
- **解释（雨）**：VMware 虚拟机默认不把宿主机的 PMU（Performance Monitoring Unit）硬件性能计数器透传给客户机。尽管尝试在 .vmx 文件中添加 `pmu.virtualize = TRUE`，但由于使用的是 AMD CPU，VMware 的 PMU 透传支持有限，最终仍无法获取硬件事件
- **行动（伞）**：在报告开头明确标注测试环境为 VMware 虚拟化，说明 instruction/cache 指标不可用的原因，改用 cycles、context-switches 等可用指标进行分析，并结合微架构原理进行理论分析

### 问题 2：stress-ng 在火焰图中显示为 [unknown]

- **事实（空）**：`perf record` 采集 stress-ng 后生成火焰图，所有内部函数全部显示为 `[unknown]`
- **解释（雨）**：Ubuntu 软件源中预编译的 stress-ng 二进制未启用帧指针（`-fno-omit-frame-pointer`），perf 默认的 fp 模式无法解析函数调用栈
- **行动（伞）**：从 GitHub 克隆 stress-ng 源码，使用 `make CFLAGS="-g -fno-omit-frame-pointer -O2"` 重新编译后替换系统版本。再跑火焰图，`[unknown]` 变成了 `stress_cpu_matrix_prod.arch_znver3.17`（矩阵乘法热点，占 99.69%）和 `queens_try`（N 皇后回溯函数，占 84%-99%）

### 问题 3：Docker 容器中 perf 无法运行

- **事实（空）**：构建 Docker 容器后，执行 `perf record` 报错 "perf not found for kernel 6.8.0-124"，提示需要安装对应内核版本的 linux-tools
- **解释（雨）**：Ubuntu 22.04 基础镜像中的 perf 版本与宿主机（6.8.0-124）内核不匹配，而 perf 是内核工具，需要与运行内核版本一致
- **行动（伞）**：在容器 entrypoint 中添加自动检测逻辑，启动时执行 `apt-get install -y linux-tools-$(uname -r)`，安装匹配当前内核的 perf 版本，同时安装缺少的共享库（libelf1、libnuma1、libtraceevent1 等）

### 问题 4：火焰图中大量 `:0;secondary_startup` 和 `:0;[unknown]`

- **事实（空）**：生成的火焰图 top 热点列表中，除了 stress-ng 外，还有大量 `:0;secondary_startup_64_no_verify` 和 `:0;[unknown]` 条目
- **解释（雨）**：`:0` 是 perf 无法将 CPU 地址解析为具体函数名的占位符。`secondary_startup` 属于内核 CPU 初始化代码，`[unknown]` 来自其他没有调试符号的进程。这在任何火焰图中都会存在——Brendan Gregg 的官方示例也有类似情况
- **行动（伞）**：重点关注占比最大的有名函数（stress-ng 热点占 87% 以上），`:0` 和 `[unknown]` 属于正常现象，在报告中如实说明即可

### 问题 5：Cache Line 测试结果的解读

- **事实（空）**：10 种步长的测试数据显示，总时间随步长增大而持续下降，没有明显的拐点
- **解释（雨）**：总时间受迭代次数影响太大（stride=1 要遍历 3200 万次，stride=512 只遍历 6 万次），不能直接看出 Cache Line 边界效应。需要计算"单次访问延迟"
- **行动（伞）**：用"单次访问延迟 = 总时间 / 迭代次数"重新计算，结果清晰显示 stride=64 时延迟从 2.75ns 跳升至 4.76ns（↑73%），验证了 Cache Line = 64 字节。同时绘制双图（总时间 + 单次访问延迟），直观展示拐点

## 七、总结

通过这个项目，我了解了 Linux perf 性能分析工具链的基本用法，学会了如何从 CPU 微架构角度分析不同负载的性能特征。从环境配置到数据采集，再到结果分析和报告撰写，整个过程让我意识到硬件和软件结合分析的重要性。后续如果有更多时间，希望能在物理机上重新测试以获取完整的硬件性能指标，以及进一步探索更多采集器类型。
